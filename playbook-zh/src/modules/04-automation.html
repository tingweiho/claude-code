自動化：全天候運作的系統

全天候運作的系統——當我不在鍵盤前時，依然在進行的工作。

三種自動化

上一個模組談的是我親手主導的工作，而這一層正好相反：在我睡覺或開會時，不需要我也能運作的工作。我在 Claude Code 上的一天，很多事都是這樣發生的。把它想成智慧家庭：三種自動化，各由不同的觸發器啟動，各能觸及不同的東西。分辨哪個是哪個，正是整套技巧的核心——它們的差別在於能觸及什麼，而大多數人只認識三者中的一種。

💡

Hook

動作感應燈。由事件觸發——一場工作階段結束、一個檔案被儲存。本機且即時。

⏰

本機 cron

灑水器定時器。由時鐘觸發。本機且 headless——沒有登入工作階段，所以也接不上任何連接的應用程式。

☁️

雲端觸發器

替你看家的人。由雲端裡的時鐘觸發，而且能應門——它可以連到 Slack、Gmail、Google Calendar 等連接的應用程式。

Hook——事件驅動、本機

hook 是事件驅動的：它監看某一個特定時刻，一到就立刻執行一段腳本。以下是我目前設定裡接上的三個。

Stop 當一場工作階段結束時，執行 log_usage.sh——把 token 與成本記到一份帳本裡。
slack_update_canvas 當專案 canvas 變動時，執行 canvas_backup_sync.sh——把它鏡像到一個本機檔案。
statusLine 執行 statusline-command.sh——我終端機裡那行 模型 / context% / git 的資訊。
CODE 
"Stop": [{ "hooks": [{
  "type": "command",
  "command": "bash ~/.claude/log_usage.sh",
  "timeout": 10
}]}]
PLAIN ENGLISH

當一場工作階段結束時（Stop 事件），執行以下動作……

……執行一個 shell 指令……

……具體來說是這支腳本，log_usage.sh……

……並給它 10 秒完成。

它會讀取我燒掉了多少 token，再把成本附加到一份試算表裡。

本機 cron——時鐘驅動、本機

cron 工作是我自己機器上的一個定時任務：在固定的時鐘時間，執行這件事。我目前唯一啟用的工作，是讓我的程式碼分支不要愈離愈遠。

CODE 
0 6,8,10,12,14,16 * * 1-5   ~/.claude/auto_rebase.sh
PLAIN ENGLISH

在早上 6 點、8 點、10 點、中午、下午 2 點、下午 4 點的第 0 分鐘……

……週一到週五……

……執行 auto_rebase.sh：把 main 分支拉進我的工作分支，讓我永遠不會落後太多。

💡
它用 merge，絕不用 rebase

儘管名字叫 rebase，這支腳本其實是用 merge 而非 rebase。merge 可以避免重寫歷史所製造出來的假衝突。這是一條得來不易的規則——存進記憶，讓我永遠不必重新學一次。

雲端觸發器——每日節奏

這些以雲端排程觸發器（CCR）的形式執行：像 cron 一樣由時鐘驅動，但在雲端、且已登入，所以能連到我連接的應用程式。其中四個，塑造了我的一整天。

1
daily-digest · 早晨

把我的專案 canvas、未讀的 Slack、行事曆、Gmail，以及 Gemini 會議紀錄，縫成一份簡報——然後在我還沒打開任何一個分頁之前，就DM 給我。

2
daily-progress-check（EOD）· 傍晚

重建我今天實際交付了什麼——從我的 GitLab 通知信裡撈出 commit 和合併請求，因為我的技術工作從來不會落在 Slack 上——然後提出追蹤器的更新建議。

3
canvas-updater · 夜間

把核准過的更新套用到專案 canvas 上。

4
gmail-auto-label · 每 2 小時

一個安靜的收件匣清潔工：依寄件者貼標籤、歸檔郵件，把雜訊封存起來。

（沒錯——Gmail 自己的篩選器不知為何對我就是不太管用。所以我讓 Claude 來做。別評判我 🫣）

它們讀什麼，才懂我的一天

我的一天散落在五六個工具裡。這些代理替我把它們收攏起來——而每一個來源，都補上其他來源補不了的缺口。

📧
GitLab 通知信

我的技術進度——commit、合併請求、審查——都待在 GitLab 裡，不在 Slack。讀它的郵件通知，就是 Claude 知道我交付了什麼的方法。

💬
Slack 訊息

掃過我的頻道與討論串，找出決策、討論，以及任何指向我的內容——這樣一句被埋住的「嘿，你能不能……」就不會溜過去。

📝
Gemini 會議紀錄

讀我通話時自動產生的 Gemini 紀錄，讓會議結論不必我打一段摘要，就能進到簡報裡。

📅
行事曆、Jira 與 Confluence

發生了什麼、接下來有什麼，以及任何提到我的票或文件。

🔗
有憑有據，不是猜的

簡報裡的每一行，都必須能追溯回代理那次執行時讀到的真實憑據——某一封具體的 GitLab 信、某一條 Slack 討論串，或某一份會議紀錄。不做合成，不發明沒發生過的進度。

追蹤器：一個 canvas，專案雷達

這一切都指向同一個 Slack canvas。它的結構，讓進度一眼就看得懂：專案依優先順序分桶，每一個都會自我說明。

一個真實的條目，去識別化後——形狀就跟我實際在用的完全一樣。

每個專案都帶著同樣的四個欄位，而每一個都回答了一個我原本得去追問的問題：

Status 它現在處在什麼狀態？——🟡 進行中、🔴 受阻、⏸️ 已降優先
ETA 它什麼時候會完成？——我正瞄準的那個日期
Next 下一個動作究竟是什麼？——這樣我重啟一個專案時不會從零開始摸索
Log 發生過什麼、各在哪天？——一條帶時間戳的軌跡，我可以據此重建脈絡
🛰️
查進度 = 打開一份文件

優先順序放在最上面，狀態與下一步寫在每一張卡片上，底下是一條附日期的紀錄。早晨摘要會讀最上面的桶；EOD 代理會把內容附加到 Log。我從不手動編輯它——我只要說一句「在〔某專案〕加上更新：……」或「把〔某專案〕的 ETA 設成……」，canvas-tracker 技能就會把它重寫好。

追一遍這個迴圈：EOD → canvas

看一個完整的循環。這是一個預設為同意的人在迴圈中流程——沉默就代表提案放行，所以即使在我最忙的日子，追蹤器也能保持最新。

這套節奏真正解決了什麼

這一切都不是為了自動化而自動化。這套每日節奏的存在，是為了消滅具體、每天都在發生的痛點。

😵‍💫

之前

  • 過夜的變動散落在 Slack、郵件、行事曆和 Jira 裡——每天早上沒有一張完整的全貌。
  • 我交付的工作都待在 GitLab 裡；到了傍晚，我已經半忘了自己究竟做了什麼。
  • 專案追蹤器漸漸過時——我老是想著要更新它。
  • 團隊和主管能不能看見進度，全看我記不記得去張貼。
😌

之後

  • 一份有憑有據的早晨簡報，由每個來源縫合而成，已經在我的 DM 裡等著。
  • EOD 直接從 GitLab 的信裡抓出當天的 commit 和合併請求。
  • 追蹤器每晚自我更新——沉默就只代表「放行」。
  • 狀態自動產生；沒有東西會從縫隙裡漏掉。
⏳
真正的勝利是注意力，不是分鐘數

我一開始這一天就已經進入狀況，結束時也不必寫一份狀態更新。這套節奏替我扛起脈絡，讓我能把專注力花在真正的工作上。
