嗨 — 我是 Ting-Wei。Claude Code 已經悄悄成為我工作的骨幹：寫 SQL、發布程式碼、畫圖表、草擬更新，並在我睡覺時把那些無聊的事跑完。

01

系統：~/.claude/ 是我的操作層

~/.claude/ 作為一個操作層——在我打出任何一個字之前，Claude Code 就已經讀過的那些檔案。

它早就知道我怎麼工作

每次我打開 Claude Code，在我輸入任何東西之前，它都會先讀過一小撮檔案。正是這些檔案，讓它早就知道我怎麼工作。

這不是我每天早上貼進去的一次性提示。我的設定是一個長期存在的操作層——一套我會做版本控制、持續改進、並從這個工作階段帶到下一個工作階段的配置。

💡
原來如此！

這份設定會在啟動時由上到下讀過一遍。所以順序與精簡都很重要——檔案愈精簡，Claude 實際上就愈會確實遵循它。

洞洞板

想像一塊工作室的洞洞板：每一件工具在牆上都有一圈漆好的輪廓，所以你一眼就能看出哪些工具在、哪些不見了。~/.claude/ 就是那塊洞洞板——每一種指示都有固定的擺放位置。

~/.claude/ 我整個操作層——一個由我擁有的資料夾
CLAUDE.md 我是誰＋該怎麼跟我說話＋我的技能索引（48 行）
skills/ 16 個按需載入的專業能力資料夾
settings.json 全域的防護欄、hook，以及資料連線
settings.local.json 那份「好，這個你可以執行」的允許清單
hooks/ 在特定事件發生時觸發的小型腳本
reference.md 我常查的 ID、檔案路徑與排程
known_issues.md 一份持續更新的修正記錄
*.sh statusline-command.sh、log_usage.sh、ci-watch.sh、auto_rebase.sh
memory/ Claude 學到、關於我的長存事實

一份精簡的索引檔、一個技能資料夾、兩個權限檔、幾個 hook，再加上一個記憶資料夾。把它當成一個程式碼庫來對待，而不是一個設定畫面。

CLAUDE.md 是一份目錄，不是一本手冊

Claude 最先讀到的這個檔案是刻意寫得很短。它告訴 Claude 該怎麼表現，然後指向真正的細節住在哪裡。

CODE 
## Communication
- Concise and direct — no filler, no trailing summaries
- Lead with the answer or action, not the reasoning

## Skills (load on demand)
- company-data-stack — Redshift / dbt / Looker / LookML
- canvas-tracker — Slack project tracker canvas ops
- data-analysis — rigor for diagnostic / exploratory work
PLAIN ENGLISH

首先它告訴 Claude 該怎麼跟我說話——簡短、答案優先。

接著它列出我的技能，每一項用一行說明它在什麼時候會啟用。

它不解釋這些技能——只是指向它們。細節住在每一個技能資料夾的裡面。

📍
我整份 CLAUDE.md 只有 48 行。

它的作用是一份向外指路的目錄，而不是一本把所有東西都裝進去的手冊。（最後一個模組會再回頭談為什麼這樣做。）

兩個權限檔，各司其職

權限被拆到兩個檔案裡，讓那份危險的清單保持簡短，而那份方便的清單可以自由地長大。

🛡️
settings.json——全域防護欄

一份破壞性指令的拒絕清單，再加上那些 hook 與資料連線。一份很少更動的共用預設。

✅
settings.local.json——允許清單

一份約 322 條我事先核准過的特定指令允許清單，這樣我才不用整天都在點「允許」。

Bash(rm -rf*) 絕不大批刪除檔案
Bash(git push --force*) 絕不改寫共用的歷史紀錄
Bash(sudo*) 絕不以管理員身分執行
🔎
為什麼要拆開？

把拒絕（全域、很少更動）和允許（本機、隨時間長大）分開，能讓那份危險的清單保持簡短，也方便稽核。

連線——Claude 接上我真正在用的工具

單靠它自己，Claude 只能說話。MCP 就是那個通用插座，讓它能在我實際用的工具裡動手做事——查詢資料倉儲、發訊息到 Slack、讀我的行事曆。一天下來，幾乎我碰到的每樣東西都接上線了。

一套標準，眾多工具

資料技術堆疊的連線在本機執行；日常 app 則是 Claude 內建的連接器。不論哪一種，背後的想法都一樣——一個標準插座，所以加進一個新工具，並不代表要重新教 Claude 怎麼跟它對話。

深入內部——一個連線，已遮蔽機密

每一個連線都只是幾行設定。這裡是資料倉儲那一個——值得認得，但你其實不太常親手寫它。

CODE 
"redshift": {
  "command": "uvx",
  "args": ["awslabs.redshift-mcp-server@latest"],
  "env": {
    "AWS_PROFILE": "<aws-account>_DataRedshift",
    "AWS_REGION": "us-west-2"
  }
}
PLAIN ENGLISH

為一個叫做 redshift 的連線命名。

uvx 即時下載並執行一個小型的輔助程式。

這份設定說明了要用哪一個帳號與區域——真正的機密住在這個檔案之外，絕不以明文存放。

⚠️
關於你正在看到的東西的一點說明

真正的設定裡裝著上線中的憑證——連線字串、用戶端密鑰、token。本課程中所顯示的一切都已遮蔽機密，而這份紀律本身就是整套配置的一部分（最後一個模組會談更多）。

02

技能庫：隨需載入的專業知識

隨需載入的專業知識——一組隨傳隨到的專家，只在任務對得上時才現身。

我從不點名召喚某位專家

我從不輸入「用 SQL 技能」。我只描述我想要什麼，對的專業知識就會自己載入進來。一個 技能 就只是一個資料夾，裡面裝著一份 SKILL.md 檔案，以及一行 描述 來決定它何時啟動。

company-data-stack/ 一個技能 = 一個資料夾
SKILL.md 完整的指示，加上開頭那行觸發描述
💡
桌上的那張名片

啟動時，Claude 只讀那些描述——專家的名片永遠擺在桌上。完整的 SKILL.md 只在任務對得上時才走進門，讓 Claude 的 脈絡視窗 保持精簡。

技能庫一覽

十六個技能，依工作性質分成三個小隊。每個小隊有自己的顏色，貫穿整個課程。

🏢

公司營運

領域小隊。company-data-stack（資料倉儲＋ BI）、canvas-tracker（Slack 專案追蹤器）、scheduled-triggers 與 internal-comms——再加上 company-slack、company-confluence、gitlab-cli。

📊

產出與分析

data-analysis、data-viz、company-slides，以及 codebase-to-course——正是打造這個頁面的那個技能——還有 pdf 和 hex。

🧰

後設

prompt-authoring、skill-creator——做出技能的技能——還有 regression-guard。

16 個技能，依工作分組——而非按字母排序。

描述本身就是觸發器

打開任何一份 SKILL.md，最上面幾行就把整個故事說完了。這裡是 frontmatter ——也就是 data-viz 技能的開頭那段。

CODE 
---
name: data-viz
description: Use whenever making a chart, plot, or
  visualization in a notebook. Trigger on "plot this",

PLAIN ENGLISH

每個技能都從一個名稱和一段描述開始。

名稱只是資料夾的標籤。

只有這段描述會一直待在 Claude 的記憶裡，直到被需要為止。

那些範例語句就是觸發線——只要你說出接近的話，完整的技能就會載入。

ℹ️
寫好一段描述，本身就是一門手藝

太籠統，技能就會一直觸發；太狹窄，它就永遠不觸發。甚至還有一個 prompt-authoring 技能，專門用來微調這件事——而且沒錯，一個 筆記本 就是資料分析師的工作畫布。

這些技能從哪裡來

這一切都不是從零打造的。整座技能庫是三種來源交織而成。

🛠️

從零自建

這些公司專屬的技能，封裝了任何下載都帶不來的內部知識——資料倉儲慣例、canvas 規則，以及公司內部的文件風格。

codebase-to-course，以及 company-slides 背後的 frontend-slides 引擎，都來自開源打造者 Zara Zhang （github.com/zarazhangrui）。我把它們 fork 下來，換上我自己的品牌與慣例。

03

使用情境：工作實際怎麼完成

工作實際上是怎麼完成的——同一個工具，四種截然不同的任務。

一天就像一場接力賽

平常的一天會橫跨四種幾乎毫無共通點的工作：算數字、畫圖表、寫給人看的東西，還有交付程式碼。同一個工具就能應付這四種——它只是切換模式，就像跑者跨越不同的地形。

把這些串起來的關鍵是：這些全都是由我親自推動的工作——雙手放在鍵盤上，一步步掌舵。那些不需要我也能跑的工作，是下一個模組的主題。這裡每一條賽道，都由不同的技能（skill）驅動——也就是上一個模組裡那些可載入的劇本。

📊
資料分析

「這個指標為什麼動了？」由 data-analysis + company-data-stack 驅動。

📈
視覺化

把一張表格變成別人真的看得懂的圖表。由 data-viz 驅動。

✍️
溝通

狀態更新、文件、簡報。由 internal-comms、company-confluence、company-slides、company-slack 驅動。

🗂️
生產力／行政雜務 ——格格不入的那一個

這是我不再親自推動的唯一一條賽道：因為太過重複，我把它交給了不需要我也能跑的機器人。這也是通往下一個模組的橋樑——daily-digest、canvas-tracker。

賽道 1——資料分析

data-analysis 技能會強制執行那些趕時間的分析師會省略的嚴謹步驟。它把每一個問題都導向五種模式之一，每種都有自己的檢查清單。

🔍

診斷型

某個指標為什麼動了？

🧭

探索型

這份新資料集裡到底有什麼？

📋

描述型

乾淨俐落地總結發生了什麼事。

⚗️

實驗型

這個 A/B 測試有效嗎？

🔮

預測型

接下來最可能發生什麼？

診斷幾乎總是從一段簡單的健全性檢查查詢開始，用的是慣用的風格——snake_case、清楚的別名、沒有花俏的子查詢。

CODE 
select
    signup_week,
    count(distinct user_id) as signups
from clean_signups
where signup_week >= date_trunc('week', current_date) - interval '8 weeks'
group by 1
order by 1
PLAIN ENGLISH

數每一週的不重複使用者人數……

……涵蓋過去 8 週……

……這樣我就能看出上週的下滑是真的，還是只是雜訊。

💡
診斷的第一條守則

在解釋下滑之前，先確認這個下滑是真的。診斷有一半的功夫，都在於抗拒大腦端給你的第一個漂亮故事。

賽道 2——視覺化

一張圖表只有在讀者一眼就看懂時才有用。data-viz 技能內建了 Cleveland–McGill 階層，也就是關於人眼能準確讀出什麼的研究成果。

📏

位置勝過角度，角度勝過面積

長條圖勝過圓餅圖——人眼讀長度遠比讀一塊扇形準確。

🚫

不用雙 Y 軸

雙 Y 軸會憑空捏造出根本不存在的關係。

🎨

用漸層，不用彩虹

連續型資料用連續型漸層，絕不用彩虹色。

其中有兩項不是我每次都重新做的決定——它們是已存好、會自動觸發的預設值。

no gridlines 預設移除，在每一個繪圖函式庫裡都一樣。
centered titles 標題預設置中，不必特別問。
💡
預設值住在記憶裡

這些偏好設定住在 Claude 的記憶裡，所以每一張圖一開始就是對的，不用我開口——我們會在第 5 個模組看到這套記憶系統。

賽道 3 與 4——溝通 + 行政雜務

寫給人看是一種獨立的語氣，而且每一種載體都有自己的技能。行政這條賽道太過重複，所以大多交給了機器人。

📝
internal-comms

用公司慣用的格式寫狀態報告、領導層更新和 FAQ。

📚
company-confluence

讓 wiki 文件落在對的空間裡、對的位置上。

🖥️
company-slides

16:9 的品牌化簡報，隨時可以上台報告。

💬
company-slack

跨頻道發文、閱讀、搜尋。

而這裡就是本模組與下一個模組之間的接縫：行政這條賽道，是我唯一停止親自推動的工作。它太過重複，於是我把它交給了不需要我也能跑的排程機器人——而這個交接，從我親自推動的工作，到自己會跑的工作，正是下一個模組的全部內容。

真正在操作工具，而不只是聊天

Claude 不只是說話——它會直接從命令列操作我真正的工具。它最常倚重的有兩個：

🦊
GitLab，透過 glab CLI

GitLab 是我的程式碼存放與交付的地方。glab 是它的命令列工具，讓 Claude 可以開合併請求、盯流程、讀審查討論串，完全不必我去碰那個網頁。

📓
Hex，透過它的 CLI

Hex 是我的分析筆記本所在的地方。它的 CLI 讓 Claude 可以開一個專案、丟進 SQL 和 Python 儲存格、跑一段查詢——把一個鬆散的問題，變成一份能運作、可分享的筆記本。

⌨️
CLI 是代理長出雙手的方式

一則聊天回覆只是文字。命令列工具讓 Claude 能動手做——開 MR、跑查詢、交付改動。下一個畫面會讓 glab 那一套動起來給你看。

04

自動化：全天候運作的系統

全天候運作的系統——當我不在鍵盤前時，依然在進行的工作。

三種自動化

上一個模組談的是我親手主導的工作，而這一層正好相反：在我睡覺或開會時，不需要我也能運作的工作。我在 Claude Code 上的一天，很多事都是這樣發生的。把它想成智慧家庭：三種自動化，各由不同的觸發器啟動，各能觸及不同的東西。分辨哪個是哪個，正是整套技巧的核心——它們的差別在於能觸及什麼，而大多數人只認識三者中的一種。

💡

Hook

動作感應燈。由事件觸發——一場工作階段結束、一個檔案被儲存。本機且即時。

⏰

本機 cron

灑水器定時器。由時鐘觸發。本機且 headless——沒有登入工作階段，所以也接不上任何連接的應用程式。

☁️

雲端觸發器

替你看家的人。由雲端裡的時鐘觸發，而且能應門——它可以連到 Slack、Gmail、Google Calendar 等連接的應用程式。

Hook——事件驅動、本機

hook 是事件驅動的：它監看某一個特定時刻，一到就立刻執行一段腳本。以下是我目前設定裡接上的三個。

Stop 當一場工作階段結束時，執行 log_usage.sh——把 token 與成本記到一份帳本裡。
slack_update_canvas 當專案 canvas 變動時，執行 canvas_backup_sync.sh——把它鏡像到一個本機檔案。
statusLine 執行 statusline-command.sh——我終端機裡那行 模型 / context% / git 的資訊。

  "type": "command",
  "command": "bash ~/.claude/log_usage.sh",
  "timeout": 10
}]}]
PLAIN ENGLISH

本機 cron——時鐘驅動、本機

cron 工作是我自己機器上的一個定時任務：在固定的時鐘時間，執行這件事。我目前唯一啟用的工作，是讓我的程式碼分支不要愈離愈遠。

CODE

PLAIN ENGLISH

在早上 6 點、8 點、10 點、中午、下午 2 點、下午 4 點的第 0 分鐘……

……週一到週五……

……執行 auto_rebase.sh：把 main 分支拉進我的工作分支，讓我永遠不會落後太多。

💡
它用 merge，絕不用 rebase

儘管名字叫 rebase，這支腳本其實是用 merge 而非 rebase。merge 可以避免重寫歷史所製造出來的假衝突。這是一條得來不易的規則——存進記憶，讓我永遠不必重新學一次。

雲端觸發器——每日節奏

這些以雲端排程觸發器（CCR）的形式執行：像 cron 一樣由時鐘驅動，但在雲端、且已登入，所以能連到我連接的應用程式。其中四個，塑造了我的一整天。

這一切都不是為了自動化而自動化。這套每日節奏的存在，是為了消滅具體、每天都在發生的痛點。

  • 過夜的變動散落在 Slack、郵件、行事曆和 Jira 裡——每天早上沒有一張完整的全貌。

  • 一份有憑有據的早晨簡報，由每個來源縫合而成，已經在我的 DM 裡等著。
  • EOD 直接從 GitLab 的信裡抓出當天的 commit 和合併請求。

⏳

我一開始這一天就已經進入狀況，結束時也不必寫一份狀態更新。這套節奏替我扛起脈絡，讓我能把專注力花在真正的工作上。

05

記憶與自我精進：回饋的循環

一次性的更正如何變成永久的習慣——讓 Claude 從工具變成隊友的回饋循環。

一位真的記得住的同事

告訴一位好員工一次——「我們一律用 merge，絕不 rebase」——你就再也不必重複。記憶正是讓 Claude 成為那位員工，而不是每週一都要重新訓練的那位的關鍵。

memory/ 約 20 個單一事實檔案——每個只放一件事
MEMORY.md 索引：每則記憶一行，每個工作階段都會載入
feedback-viz-no-gridlines.md 我喜歡的工作方式——更正與已確認的做法
project-pro-plan-cutout.md 程式碼無法告訴你的、持續進行中的脈絡
reference-confluence-map.md 指向各種 ID、文件、儀表板的指標

約 20 個單一事實檔案。每個檔案一件事——好新增，也好淘汰。

一次更正如何變成永久

我第一次更正 Claude 時，它到下個工作階段就忘了。現在不會了——因為那次更正被寫進了像這樣的一個小檔案。

每一個繪圖函式庫。原因：更乾淨、墨水更少。
PLAIN ENGLISH

一個 frontmatter name——一個簡短的 slug，讓我能連結到這則確切的記憶。

一段 description——Claude 會掃描的那一行，用來判斷這則記憶此刻是否相關。

事實本身，再加上原因，這樣它才會帶著判斷力套用——而不是盲目照做。

說一次，永遠套用

我只說過一次「拿掉格線」。現在每張圖一開始就是那樣——橫跨 Matplotlib、Plotly，任何東西都一樣。

每一次更正都讓下一個工作階段更敏銳一點。這是一個循環——我用得愈多，這套設定就愈好。

06

最佳實踐與血淚教訓

第一線的實戰筆記 —— 都是我當初開始前希望有人能先告訴我的事，寫給下一個接手的人。

CLAUDE.md 是一份目錄，不是一本說明書

如果你只想從我的設定裡抄一樣東西，請抄習慣 —— 而不是檔案。而第一個習慣就是這個：你的 CLAUDE.md 是一份目錄，不是一本教科書。

身分與偏好放在最上方，穩定而很少更動。底下放的是一份索引 —— 每個技能一行 —— 指向真正細節所在的地方。

## Communication     （要怎麼跟我溝通）
## Coding            （我的慣例）
## Skills            （索引 —— 每個一行，指向外部）
## Reference files   （指標，不是內容）
PLAIN ENGLISH

身分與偏好放最上方 —— 穩定，很少改動。

我希望 Claude 怎麼說話，以及我寫程式遵循的慣例。

接著是一份技能索引：每個一行，沒有細節。

細節住在技能資料夾與記憶裡 —— CLAUDE.md 只負責把路指過去。

💡
我的只有 48 行

每一行在每次工作階段開始前都會被讀過一遍 —— 那是花掉的注意力。一份臃腫的 CLAUDE.md 會讓 Claude 變差，而不是變好。把它維持成一份目錄。

能搬到任何一套設定的那些習慣

這套設定之所以行得通，靠的是幾個原則，而不是哪一段聰明的腳本。這四個習慣能搬到任何一套 Claude Code 設定、任何一個團隊。

🎯

按需載入

把流程放進技能裡，而不是放進 CLAUDE.md。脈絡是一筆預算 —— 只在相關時才花用。

🍴

用 fork 起家，別從零造起

從一個好的開源技能起步（例如 Zara Zhang 的），改頭換面成自己的，再用 skill-creator 慢慢長出你自己的。

🧠

把修正寫下來

一行字的記憶檔案，能把「我得講第二次」變成「我從此再也不用講」。

🤖

把無聊的事自動化，把人留在迴圈裡

預設核准的流程（像我那套下班 → canvas 的迴圈）讓事情持續推進，又不會脫韁演出。

先計畫，再動工

只要比一次小修改更大的事，我都會逼 Claude 先計畫，再碰任何一個檔案。在計畫模式裡，它探索問題、提出做法，然後等我核准 —— 這樣在改動還免費的時候，我就能把它導回正軌。

好處是槓桿效應：在一段話的計畫裡抓到一個錯誤假設，只花幾秒；等 Claude 已經改寫了十個檔案才抓到，得賠上一整個下午。

全部六個模組都是在計畫模式裡界定範圍，而我是在寫下任何一行 HTML 之前就核准了大綱。在紙上重塑很便宜；在程式碼裡重塑很昂貴。

遮蔽，是分享的代價

有幾道指令就是怎麼算都不值得冒那個險，所以它們被全域封鎖 —— 不管被怎麼要求，Claude 都不能執行。這些護欄住在 settings 裡，套用到每一段工作階段。
