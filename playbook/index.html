01

The System

~/.claude/ as an operating layer — the files Claude Code reads before I type a word.

It already knows how I work

Every time I open Claude Code, it reads a handful of files before I type anything. Those files are why it already knows how I work.

This isn't a one-off prompt I paste in each morning. My config is a standing operating layer — a setup I version, improve, and carry from session to session.

💡
Aha!

The config is read top-to-bottom at startup. So order and brevity matter — the leaner the file, the more reliably Claude actually follows it.

The pegboard

Think of a workshop pegboard: every tool has a painted outline on the wall, so you see at a glance what's there and what's missing. ~/.claude/ is that pegboard — a fixed place for every kind of instruction.

~/.claude/ My whole operating layer — one folder I own
CLAUDE.md Who I am + how to talk to me + an index of my skills (48 lines)
skills/ 16 folders of on-demand expertise
settings.json Global guardrails, hooks, and data connections
settings.local.json The "yes, you may run this" allow-list
hooks/ Small scripts that fire on events
reference.md IDs, file paths, schedules I look up often
known_issues.md A running fixes log
*.sh statusline-command.sh, log_usage.sh, ci-watch.sh, auto_rebase.sh
memory/ Durable facts Claude has learned about me

A lean index file, a skills folder, two permission files, some hooks, and a memory folder. Treat it like a codebase, not a settings screen.

CLAUDE.md is a directory, not a manual

The file Claude reads first is short on purpose. It tells Claude how to behave, then points at where the real detail lives.

CODE 
## Communication
- Concise and direct — no filler, no trailing summaries
- Lead with the answer or action, not the reasoning

## Skills (load on demand)
- company-data-stack — Redshift / dbt / Looker / LookML
- canvas-tracker — Slack project tracker canvas ops
- data-analysis — rigor for diagnostic / exploratory work
PLAIN ENGLISH

First it tells Claude how to talk to me — short, answer-first.

Then it lists my skills with one line each on when they fire.

It doesn't explain the skills — it just points to them. The detail lives inside each skill folder.

📍
My whole CLAUDE.md is 48 lines.

It works as a table of contents that points outward, not a manual that holds everything. (We'll come back to why in the last module.)

Two permission files, one job each

Permissions are split across two files so the dangerous list stays short and the convenient list can grow freely.

🛡️
settings.json — global guardrails

A deny-list of destructive commands, plus the hooks and the data connections. A shared default that rarely changes.

✅
settings.local.json — the allow-list

An allow-list of ~322 specific commands I've pre-approved, so I'm not clicking "allow" all day.

Bash(rm -rf*) Never bulk-delete files
Bash(git push --force*) Never rewrite shared history
Bash(sudo*) Never run as administrator
🔎
Why split them?

Separating deny (global, rarely changes) from allow (local, grows over time) keeps the dangerous list short and auditable.

Under the hood — one connection, redacted

Each connection is just a few lines of config. Here's the warehouse one — useful to recognize, but you never write it by hand often.

CODE 
"redshift": {
  "command": "uvx",
  "args": ["awslabs.redshift-mcp-server@latest"],
  "env": {
    "AWS_PROFILE": "<aws-account>_DataRedshift",
    "AWS_REGION": "us-west-2"
  }
}
PLAIN ENGLISH

Names a connection called redshift.

The settings say which account and region — the actual secrets live outside this file, never in plain text.

⚠️
A note on what you're seeing

Real configs hold live credentials — connection strings, client secrets, tokens. Everything shown in this course is redacted, and that discipline is itself part of the setup (more in the last module).

Lane 1 — Data analysis

The data-analysis skill enforces the rigor a rushed analyst skips. It routes every question into one of five modes, each with its own checklist.

🔍

Diagnostic

Why did a metric move?

🧭

Exploratory

What is even in this new dataset?

📋

Descriptive

Summarise what happened, cleanly.

⚗️

Experimental

Did the A/B test work?

🔮

Predictive

What is likely to happen next?

A diagnostic almost always starts with a simple sanity-check query, written in the house style — snake_case, clear aliases, no clever subqueries.

CODE 
select
    signup_week,
    count(distinct user_id) as signups
from clean_signups
where signup_week >= date_trunc('week', current_date) - interval '8 weeks'
group by 1
order by 1
PLAIN ENGLISH

Count distinct users per week...

...for the last 8 weeks...

...so I can see if last week's drop is real or just noise.

💡
The first rule of diagnostics

Confirm the drop is real before explaining it. Half of diagnostics is resisting the first tidy story your brain offers.

Lane 2 — Visualization

A chart is only useful if a reader gets it in a glance. The data-viz skill bakes in the Cleveland–McGill hierarchy, the research on what the human eye reads accurately.

📏

Position beats angle beats area

Bars over pies — the eye reads length far better than a slice.

🚫

No dual Y-axes

A dual Y-axis invents relationships that aren't there.

🎨

Ramp, not rainbow

Sequential data gets a sequential ramp, never a rainbow.

Two of these aren't decisions I re-make each time — they're saved defaults that fire automatically.

no gridlines Removed by default, in every charting library.
centered titles Titles sit centered, by default, without asking.
💡
Defaults live in memory

These preferences live in Claude's memory, so every chart starts correct without me asking — we'll see the memory system in Module 5.

Meet the runners: a branch opens the MR; a CI/CD pipeline tests it; ci-watch.sh babysits the pipeline; Bugbot leaves a review; and finally I merge into the protected main copy.

Rather than refreshing a web page for ten minutes, ci-watch.sh collapses the whole noisy pipeline into a single exit code.

CODE 
# ci-watch exit codes
0   green — pipeline passed, no open threads
1   failed — prints the last 120 lines of the failing job
2   timeout — 40-minute cap hit
4   green, but review threads still open
PLAIN ENGLISH

Instead of refreshing a web page, I get one number.

1 drops me straight onto the failing log.

4 means tests pass but a human or green still isn't enough: a review thread from Bugbot wants changes — not done yet.

💡
A script can encode "done"

The exit codes encode a definition of done: green is necessary but not sufficient. Open threads still block the merge.

04

The Automation Layer

The always-on system — the work that happens when I am not at the keyboard.

Three kinds of automation

Where the last module was hands-on work I drive, this layer is the mirror image: work that runs without me — while I am asleep or in meetings. A lot of my Claude Code day happens this way. Think of it like a smart home: three kinds of automation, each tripped by a different trigger, each able to touch different things. Knowing which is which is the whole skill — they differ in what they can touch, and most people only know one of the three.

💡

Hooks

Motion-sensor lights. Fire on an event — a session ends, a file is saved. Local and instant.

⏰

Local cron

A sprinkler timer. Fires on a clock. Local and headless — no login session, so no connected apps.

☁️

Cloud triggers

A house-sitter. Fires on a clock in the cloud, and can answer the door — it reaches connected apps like Slack, Gmail, and Calendar.

Hooks — event-driven, local

A hook is event-driven: it watches for one specific moment and runs a script the instant it arrives. Here are the three wired into my setup right now.

Stop When a session ends, run log_usage.sh — logs tokens and cost to a ledger.
slack_update_canvas When the project canvas changes, run canvas_backup_sync.sh — mirrors it to a local file.
statusLine Runs statusline-command.sh — the model / context% / git line in my terminal.
CODE 
"Stop": [{ "hooks": [{
  "type": "command",
  "command": "bash ~/.claude/log_usage.sh",
  "timeout": 10
}]}]
PLAIN ENGLISH

When a session ends (the Stop event), do the following...

...run a shell command...

...specifically this script, log_usage.sh...

...and give it 10 seconds to finish.

It reads how many tokens I burned and appends the cost to a spreadsheet.

Local cron — clock-driven, local

A cron job is a timed task on my own machine: at a fixed clock time, run this. My one active job keeps my code branches from drifting.

CODE 
0 6,8,10,12,14,16 * * 1-5   ~/.claude/auto_rebase.sh
PLAIN ENGLISH

At minute 0 of 6am, 8, 10, noon, 2pm, and 4pm...

...Monday through Friday...

...run auto_rebase.sh: pull the main branch into my working branches so I never drift far behind.

💡
It merges; it never rebases

Despite the name, the script merges rather than rebases. Merging avoids the false conflicts that history-rewriting creates. A hard-won rule — saved to memory so I never relearn it.

Cloud triggers — the daily rhythm

These run as cloud scheduled triggers (CCR): clock-driven like cron, but in the cloud and signed in, so they can reach my connected apps. Four of them shape my whole day.

1
daily-digest · morning

Stitches my project canvas, unread Slack, calendar, Gmail, and Gemini meeting notes into one briefing — then DMs it to me before I've opened a single tab.

2
daily-progress-check (EOD) · evening

Reconstructs what I actually shipped — pulling commits and merge requests from my GitLab notification emails, since my technical work never lands in Slack — then proposes tracker updates.

3
canvas-updater · night

Applies the approved updates to the project canvas.

4
gmail-auto-label · every 2 hours

A silent inbox janitor: labels and files mail by sender, archives the noise.

(Yes — Gmail's own filters never quite work for me, somehow. So I let Claude do it. Don't judge me 🫣)

What they read to know my day

My day is scattered across half a dozen tools. The agents gather it so I don't have to — and each source fills a gap the others can't.

📧
GitLab notification emails

My technical progress — commits, merge requests, reviews — lives in GitLab, not Slack. Reading its email notifications is how Claude knows what I shipped.

💬
Slack messages

Scans my channels and threads for decisions, discussions, and anything aimed at me — so a buried "hey, can you…" doesn't slip past.

📝
Gemini meeting notes

Reads the auto-generated Gemini notes from my calls, so meeting outcomes reach the brief without me typing a recap.

📅
Calendar, Jira & Confluence

What happened, what's coming next, and any tickets or docs that mention me.

🔗
Grounded, not guessed

Every line in a brief must trace back to a real artifact the agent read that run — a specific GitLab email, Slack thread, or meeting note. No synthesis, no inventing progress that didn't happen.

What the rhythm actually solves

None of this is automation for its own sake. The daily rhythm exists to kill specific, daily pain.

😵‍💫

Before

  • Overnight changes scattered across Slack, email, calendar, and Jira — no single picture each morning.
  • My shipped work lived in GitLab; by evening I'd half-forget what I'd actually done.
  • The project tracker drifted stale — I kept meaning to update it.
  • Team and manager visibility depended on me remembering to post.
😌

After

  • One grounded morning brief, stitched from every source, waiting in my DM.
  • EOD captures the day's commits and merge requests straight from GitLab emails.
  • The tracker updates itself nightly — silence just means "ship it."
  • Status is automatic; nothing slips through the cracks.
⏳
The real win is attention, not minutes

I start the day already oriented and end it without writing a status update. The rhythm carries the context so I can spend my focus on the actual work.

How a correction becomes permanent

The first time I corrected Claude, it forgot by the next session. Now it doesn't — because the correction gets written to a small file like this one.

CODE 
---
name: feedback-viz-no-gridlines
description: No gridlines + centered titles by default
metadata:
  type: feedback
---
No gridlines + centered titles by default; applies
to every plotting library. Why: cleaner, less ink.
PLAIN ENGLISH

A frontmatter name — a short slug so I can link to this exact memory.

A description — the one line Claude scans to decide if this memory is relevant right now.

The fact itself, plus the why, so it's applied with judgment — not blindly.

💡
Said once, applied forever

I said "drop the gridlines" once. Now every chart starts that way — across Matplotlib, Plotly, anything.

06

Best Practices & Hard-Won Lessons

Day-one field notes — the things I wish someone had told me before I started, written for the next person.

CLAUDE.md is a directory, not a manual

If you copy one thing from my setup, copy the habits — not the files. And the first habit is this: your CLAUDE.md is a table of contents, not a textbook.

Identity and preferences sit up top, stable and rarely touched. Below them lives an index — one line per skill — that points outward to where the real detail lives.

CODE 
## Who I am          (1-2 lines)
## Communication     (how to talk to me)
## Coding            (my conventions)
## Skills            (index — one line each, points outward)
## Reference files   (pointers, not content)
PLAIN ENGLISH

Identity and preferences up top — stable, rarely changes.

How I want Claude to talk, and the conventions I code by.

Then an index of skills: one line each, no detail.

The detail lives in the skill folders and memory — CLAUDE.md just routes to it.

💡
Mine is 48 lines

Every line is read before each session — that's attention spent. A bloated CLAUDE.md makes Claude worse, not better. Keep it a table of contents.

The habits that transfer to any setup

The setup works because of a few principles, not because of any one clever script. These four travel to any Claude Code setup, on any team.

🎯

Load on demand

Put procedures in skills, not in CLAUDE.md. Context is a budget — spend it only when relevant.

🍴

Fork, don't build from zero

Start from a good open-source skill (e.g. Zara Zhang's), rebrand it, then grow your own with skill-creator.

🧠

Write down corrections

One-line memory files turn "I had to say it twice" into "I never say it again."

🤖

Automate the boring, keep humans in the loop

Default-approve flows (like my end-of-day → canvas loop) keep things moving without going rogue.

Plan first, then build

For anything bigger than a quick edit, I make Claude plan before it touches a file. In plan mode it explores the problem, proposes an approach, and waits for my approval — so I can redirect it while changes are still free.

The payoff is leverage: catching a wrong assumption in a one-paragraph plan costs seconds; catching it after Claude has rewritten ten files costs the afternoon.

Redaction is the price of sharing

A few commands are just never worth the risk, so they're denied globally — Claude can't run them no matter what it's asked. These guardrails live in settings and apply to every session.

Bash(rm -rf*) Denied globally — never recursively force-delete files
Bash(git push --force*) Denied globally — never overwrite shared history with a force-push
Bash(sudo*) Denied globally — never run commands as the all-powerful admin user
⚠️
Treat anything you share like a screenshot that lives forever

A shareable setup must never leak live secrets — connection strings, account numbers, client secrets, tokens. This very course was redaction-audited before it left my machine.

Keep your failures — they're the reusable part

The most reusable part of a setup isn't the list of what worked. It's the list of what didn't.
